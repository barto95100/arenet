# Arenet — Project Context for Claude Code

## What is Arenet

Arenet is a **homelab-friendly reverse proxy with integrated security**, built on **Caddy v2 as the embedded HTTP engine**. The goal is to combine the production-grade HTTP capabilities of Caddy with a homelab-friendly admin UI (inspired by Zoraxy and NGINX Proxy Manager), plus native security integrations (WAF, IP reputation) that are usually only found in commercial products.

**Status**: Early POC, single-developer project. Not production ready.

**License**: AGPL-3.0. Every Go source file MUST include the AGPL header (see below).

## Tech Stack

### Backend (Go 1.25+)

- **HTTP engine**: Caddy v2 (`github.com/caddyserver/caddy/v2`) embedded as library, NOT as external process
- **WAF**: `github.com/corazawaf/coraza-caddy/v2`
- **CrowdSec**: `github.com/hslatman/caddy-crowdsec-bouncer`
- **Storage**: BoltDB (`go.etcd.io/bbolt`), file-based, single binary
- **WebSocket**: `github.com/gorilla/websocket`
- **Routing internal API**: stdlib `net/http` + `gorilla/mux` (for path params)
- **Logging**: `log/slog` (stdlib structured logging)
- **Config**: env vars + flags via `flag` stdlib

### Frontend

- **Framework**: SvelteKit (Svelte 5) with TypeScript
- **Build**: `@sveltejs/adapter-static` (output is plain HTML/CSS/JS, embedded in Go binary via `embed.FS`)
- **Styling**: Tailwind CSS
- **Topology graph**: D3.js
- **State management**: Svelte stores (no Redux/Pinia)

## Coding Conventions

### Go

- **Format**: `gofmt -s` clean, must pass `go vet ./...`
- **Linter**: code should pass `staticcheck ./...` without warnings
- **Tests**: each package should have `_test.go` files; aim for >50% coverage on core packages
- **Errors**: wrap errors with `fmt.Errorf("context: %w", err)`, never use `panic` outside of `main`
- **Logging**: use `slog` everywhere, never `fmt.Println` for runtime logs
- **No magic numbers**: constants at package level, well-named
- **Context propagation**: every function that does I/O takes `ctx context.Context` as first arg
- **No global state**: dependencies passed via constructors

### Frontend

- **TypeScript strict mode** enabled
- **Components**: PascalCase filenames (`RouteList.svelte`)
- **Stores**: in `lib/stores/` directory
- **API client**: centralized in `lib/api/client.ts`
- **No CSS frameworks beyond Tailwind**
- **Accessibility**: all interactive elements must have proper ARIA labels

### Git

- **Branch naming**: `feature/short-description`, `fix/short-description`
- **Commits**: imperative mood ("Add X" not "Added X"), max 72 chars subject
- **Co-author**: when Claude Code makes substantial contributions, include in commit footer

## Phase 1 Scope (current)

Build a working POC that demonstrates the architecture is viable:

1. Embedded Caddy starts and accepts a basic config from BoltDB
2. REST API to CRUD routes (host + upstream + TLS + WAF on/off)
3. Minimal admin UI (SvelteKit) to manage routes via the REST API
4. WebSocket pushing per-route req/s metrics every 1 second
5. Topology dashboard with D3.js graph + animated particles based on traffic
6. Coraza WAF enabled on routes when configured
7. Single binary deployment (`go build`), Docker image as bonus

## Phase 2 and beyond (not now)

- CrowdSec full integration with admin UI
- Advanced health checks (status code, body match, headers)
- Rate limiting per route
- Multi-tenant / multi-admin
- Backup/restore via UI
- HTTP/3 explicit support
- Statistics dashboard (Prometheus integration)

## What Claude Code Should Do

When generating code for this project:

1. **Always include the AGPL header** at the top of every Go and TypeScript file
2. **Use the project structure** described above; don't invent new directories without justification
3. **Generate working, runnable code** — not pseudo-code. Each file should compile.
4. **Compile-check often**: after generating a package, suggest running `go build ./...` or `go vet ./...`
5. **Write tests** when generating non-trivial logic. Test files alongside source files (`foo_test.go`).
6. **Comment public APIs**: every exported function/type needs a Go doc comment
7. **No dead code**: don't generate stubs or `// TODO` placeholders that don't compile
8. **Match Caddy's actual API**: when configuring Caddy, generate valid Caddy JSON config (refer to https://caddyserver.com/docs/json/)
9. **Be conservative with deps**: prefer stdlib, then well-known libs. Avoid obscure dependencies.

## What Claude Code Should NOT Do

- Don't generate Docker subprocess code to launch Caddy externally — Caddy must be embedded as a library
- Don't use `panic()` in non-test code outside of `main`
- Don't copy code from Zoraxy directly (different license/architecture)
- Don't introduce React, Vue, or other frontend frameworks — we use SvelteKit
- Don't suggest non-AGPL-compatible licenses or deps with incompatible licenses
- Don't fabricate API endpoints or library functions — verify they exist in the actual lib
