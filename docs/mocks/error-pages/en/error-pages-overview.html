Sanitizer compliance

  • ✓
    No forbidden element — no <main>, <section>, <article>, <header>, <footer>, <nav>, <svg>, <button>, <script>, <iframe>, <form>, <input>
  • ✓
    No inline event handler — no onclick, onload, onerror, nor href="javascript:"
  • ✓
    Structure — <div class="content"> with z-index:5 instead of <main>
  • ✓
    Actions — <a href="/console"> styled as a button, never <button>
  • ✓
    Animations — CSS @keyframes only, no JS, no SVG. The 12 particles are hardcoded <div> elements with :nth-child(n) for their vectors
  • ✓
    CSS — <style> block in <head> preserved via AllowUnsafe(true), supports @keyframes, CSS vars, @media, oklch(), pseudo-elements
  • ✓
    Fonts — system stacks (system-ui, sans-serif) — no Google Fonts CDN, to avoid nofollow injection by the sanitizer
  • ✓
    Color fallback — no background-clip:text + -webkit-text-fill-color:transparent on the main code (text would stay invisible if the CSS var fails to load)

Using them in Arenet

Arenet serves these pages through its embedded Caddy, configured via automatically generated JSON — there is no Caddyfile to edit. These HTML files are templates meant to be pasted directly into the admin UI.

To use one: Settings → Error Pages, create a template, then paste the HTML body of each file into the matching status-code tab (401 / 403 / 404 / 429 / 500 / 502 / 503 / 504).

The placeholders ({http.error.id}, {http.request.uuid}, etc.) are substituted by Caddy directly at serve time — no extra configuration is needed.

Then attach that template to a route (or set it as the catch-all default) from the route editor.
