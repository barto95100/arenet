# v1.3.0-step-u — Cert events in Activity log (2026-06-06)

Late closeout for Step U, shipped 2026-06-06. Aligns
discipline with the Step T release-notes pattern and
unblocks the Step V spec freeze §9 reference. Step U
surfaces ACME certificate lifecycle events
(`cert_obtained`, `cert_failed`, `cert_ocsp_revoked`) in
the operator-facing Activity log page (formerly "Security
events"). Builds on Step T's forward-compat seam
(T.1 AC #18, ghost-subscriber test) — a single
subscription block in `caddymgr` now fans out to both the
`CertInfoTracker` (Step T) and the new `CertEventSink`
(Step U).

## Highlights

- **New `cert_event` table** in `metrics.db`, schema v5,
  90-day retention matching the Let's Encrypt cert
  lifecycle. Schema-versioned migration applied at boot.
- **Channel + batcher sink pattern** mirroring the
  existing `waf` / `throttle` / `decision` sinks, with a
  non-blocking publish path so caddymgr's event handler
  is never back-pressured by storage I/O.
- **`GET /api/v1/observability/cert-events`** — full
  filter support (`limit`, `since`, `until`, `level`,
  `search`) and AC #13 degraded-mode behavior carried
  forward from Step T (nil reader → `200` with
  `degraded:true`, never `5xx`).
- **Activity log as 4th aggregator source** — cert events
  join the existing unified table additively, no UI
  rework. The Step R "Security events" page already had
  the table; Step U widens its scope.
- **Filter textbox naturally covers** "cert",
  "certificat", "Let's Encrypt", "DNS-01", and the cert
  domain itself — no new filter UI required.
- **Page rename**: "Security events" → "Activity log" to
  reflect the widened scope (cert events are not
  security events; the page now aggregates all
  operationally interesting timeline data).
- **5 boot-time signals** (HF4 pattern fully
  generalized): `certinfo_reconcile`, `cert_event_sink`,
  `sink_subscribed`, `tracker_purger_present`,
  `cert_event_reader_present` — visible at startup so any
  future wire-up regression is immediately diagnosable
  via `journalctl`.

## Backend changes

- **New `cert_event` table** (schema v5) in `metrics.db`
  with columns `ts`, `event_type`, `level`, `domain`,
  `issuer`, `acme_method`, `error_message`, `details_json`.
  Indexed on `ts` for the time-range queries the API does.
  Retention loop prunes rows older than 90 days on the
  same ticker as the WAF / throttle sinks.
- **New `cert_event_sink` package** — channel-fed
  batcher writing into the new table. Shape mirrors
  `waf_sink`, `throttle_sink`, `decision_sink`. Drop-
  metrics counter emitted on channel-full to surface
  back-pressure without blocking caddymgr.
- **`cert_event_adapter` in `caddymgr`** —
  `Tracker.Subscribe(handler)` fans the existing
  Caddy-event stream out to both certinfo (Step T) and
  the new sink. `EventCertOCSPRevoked` dispatched as a
  synthetic event ahead of the certmagic upstream not
  emitting it natively (verified empirically against
  vendored sources, per CLAUDE.md §Empirical
  verification).
- **`GET /api/v1/observability/cert-events` handler**
  registered under the admin-auth chi group, reading
  through a small `CertEventReader` interface for testability
  + the AC #13 degraded-mode contract.
- **Boot-time visibility**: `cmd/arenet` logs
  `cert_event_sink=<present>`,
  `sink_subscribed=<present>`,
  `cert_event_reader_present=<bool>` at boot. Combined
  with Step T's `purger_present` and Step T HF4's
  `certinfo_reconcile=<present>`, that's 5 boot signals
  generalized from the HF4 single-line pattern.

## Frontend changes

- **`fetchCertEvents` + `CertEvent` type** in
  `lib/api/security.ts` — single typed wrapper around the
  new endpoint, matching the existing `fetchWafEvents` /
  `fetchThrottleEvents` shape.
- **Cert events wired into the Activity log aggregator**
  in `routes/logs/+page.svelte` as a 4th source. The
  existing unified table renders them directly through
  the same row component — no new column, no new badge,
  no new badge-variant.
- **Page title + subtitle renamed** to "Activity log" /
  the matching subtitle string. The route URL `/logs` is
  unchanged.
- **Existing `--status-info / --status-warn /
  --status-down` palette** carries the cert-event level
  rendering (locked vocabulary from Step R, migrated to
  all consumers as part of the smoke-time hotfix below).

## Known limitations

- **`cert_obtaining` is not persisted** (§3.3). The
  obtaining lifecycle phase is intentionally fire-and-
  forget; persisting it would create churn rows that add
  no operational signal. The future WS-push channel will
  surface the obtaining state in real time without
  hitting `cert_event`.
- **`cached_managed_cert` / `cached_unmanaged_cert` are
  not persisted** (§3.5). These fire on every boot for
  every cached cert and would dominate the table without
  signaling anything actionable. Boot-time reconcile
  already covers the "what's in the pool" question via
  certinfo.
- **`EventCertRemoved` is not persisted** (§3.8). The
  synthetic event (Step T HF3, commit `e4177e4`) exists
  only to drive the tracker purge; certmagic / Caddy
  v2.11.3 emit no upstream cert-removal event, so a
  `cert_event` row would record a UI-driven mutation,
  not a runtime fact.
- **Polling only (v1)** (§3.7). The Activity log page
  refreshes via the existing aggregator polling cadence.
  WS-push for cert events is queued behind the broader
  observability-WS-channel work (`#R-LOGS-ws-push`).
- **Log level is INFO / ERROR only** (§3.6). There is no
  `WARN` cert-event today; the underlying certmagic
  events don't model it. `#R-LOGS-error-level` tracks the
  warn-level workaround surface, deferred.
- **Step T `#R-CERTS-force-renew` remains deferred**
  (carried from v1.2.0-step-t). The Step U sink does NOT
  add a force-renew event source.
- **`#R-CERTS-zerossl-backup` and `#R-CERTS-multi-dns`**
  remain backlog items; Step U makes them visible in the
  Activity log when they ship but does not implement
  them.

## Migration notes

- **Schema migration** runs automatically at boot
  (`metrics.db` schema v4 → v5). New table only; no
  ALTER on existing tables. Rollback path: drop the
  `cert_event` table and reset the schema version
  pragma; Step T behavior is unaffected.
- **`/logs` URL is unchanged** — operators landing on
  bookmarks see the renamed page with the widened scope.
- **No wire change to existing endpoints** — the new
  `GET /api/v1/observability/cert-events` is purely
  additive. WAF / throttle / decision endpoints retain
  their v1 vocabulary.

The U.7 smoke surfaced a long-standing token gap
inherited from pre-Step-R: the mock-vocabulary shortcuts
(`--ok`, `--warn`, `--bad`, `--down`, `--info`) were
never defined globally in `tokens.css`, yet 54+ usages
referenced them across 12 files. `/topology` survived via
inline fallbacks; `/logs` rendered the "live" pill in
white because its `.status-pill` had no fallback. The
hotfix completes the Step R intent (`tokens.css` line 78:
"OKLCH equivalents of the mock's --ok/--warn/--bad/
--info") by pointing all consumers at the canonical
`--status-up / --status-warn / --status-down /
--status-info` palette. Pre-existing bug; not introduced
by Step U.
