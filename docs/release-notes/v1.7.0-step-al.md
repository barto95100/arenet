# v1.7.0-step-al — Alerting subsystem (2026-06-16)

Step AL ships the **first proactive notification layer**
in Arenet. Previous observability features (Step L
metrics, Step M WAF events, Step N CrowdSec decisions,
Step T cert lifecycle, Step V geo events, Step W
country blocks) all share the same posture : Arenet
records signals, the operator reads them via dashboard
or activity logs. Step AL flips that polarity — the
operator declares **rules** ("alert if WAF blocks > 50
in 5 min", "alert if any cert expires in < 14 days",
"alert if the CrowdSec component degrades") and Arenet
proactively dispatches the alerts to operator-configured
**channels** (webhook, email).

The subsystem is end-to-end native: storage, evaluation,
fan-out, persistence, and a three-tab UI all live in the
single arenet binary. No external Alertmanager, no
sidecar, no shell glue.

Use case: a homelab operator configures one Slack webhook
+ one email channel, then sets up three rules (WAF rate,
cert expiry, CrowdSec health). They forget about it.
Three weeks later, a cert renewal silently fails — Arenet
fires the rule at T-14d, the operator gets a Slack
notification, fixes the DNS challenge, re-renews. Without
Step AL, the operator would have noticed only when the
TLS handshake started failing in production.

## Highlights

- **Webhook + email channels** (AL.1.b, commit `9c22b36`).
  Operator-configured HTTP POST or SMTP delivery. Webhook
  supports custom headers, template body, timeout. Email
  supports STARTTLS / implicit TLS / no encryption, auth,
  multi-recipient (To/Cc/Bcc with BCC correctly absent
  from DATA headers).
- **Threshold + state evaluators** (AL.2.a, commit
  `b1c67b9`). Threshold compares numeric source values
  via 6 operators (`>`, `>=`, `<`, `<=`, `==`, `!=`).
  State matches string source values against a literal.
- **3 bootstrap sources** (AL.2.a). `waf_event_rate`
  counts `waf_event` rows in a sliding window;
  `cert_expiry` reads `certinfo.Tracker` for days-until-
  NotAfter; `system_health` reads the AL.3a checker
  for global or per-component status.
- **Watcher runtime** (AL.2.b, commit `8fd4cc9`). One
  goroutine polls every 30 seconds. Per-`(rule, channel)`
  cooldown LRU prevents alert spam (30 s minimum,
  24 h maximum, 5 min default).
- **Rule CRUD HTTP layer** (AL.3b, commit `4883dca`).
  POST / GET / PUT / DELETE under `/api/v1/settings/
  alerting/rules` with 3 new audit actions
  (`alert_rule_created` / `_updated` / `_deleted`,
  audit count 46→49).
- **`/system/health` endpoint** (AL.3a, commit `f350640`).
  5-component check (Caddy, BoltDB, metrics, CrowdSec,
  certmagic), exposes a stable JSON shape for external
  monitoring scrape. Doubles as the `system_health`
  source for the alerting watcher.
- **History tab + persistence** (AL.4.a, commit
  `329dbbd`). Every dispatch persists one
  `alert_event` row (event_id UUID, rule snapshot,
  channels fired/failed JSON). `GET /api/v1/
  observability/alert-events` exposes the data with
  cursor pagination + severity / rule / category /
  date filters. 30 d retention.
- **Frontend Alerting page** (AL.4.b.1+.2+.3, commits
  `591f381` / `ada0b66` / `ab252e5`). Sidebar entry
  "Alerting" in the Sécurité section. Three-tab page
  (Canaux / Règles / Historique) with deep-link URL
  hashes. Channel modal with kind switching (webhook /
  email), password preserve-on-omit. Rule modal with
  dynamic source/eval sub-forms, channel multi-select,
  inline test result. History tab with debounced
  filters + load-more pagination.
- **Operator documentation** (AL.5, this commit).
  500-line guide in French at `docs/alerting.md` with
  step-by-step setup walkthroughs, troubleshooting,
  architecture diagram, and limitations.

## Backend architecture

```
┌─────────────────┐    poll 30s    ┌──────────────┐
│   AlertRule     │───────────────▶│   Watcher    │
│   (BoltDB)      │                │  (goroutine) │
└─────────────────┘                └──────┬───────┘
                                          │ source.Read
                                          ▼
                                   ┌──────────────┐
                                   │  Evaluator   │
                                   │  (threshold  │
                                   │   / state)   │
                                   └──────┬───────┘
                                          │ fire = true
                                          ▼
┌─────────────────┐                ┌──────────────┐
│  Cooldown LRU   │◀──────skip──── │  Dispatcher  │
│  (in-memory)    │                │  fan-out     │
└─────────────────┘                └──────┬───────┘
                                          │ per channel
                                          ▼
                          ┌───────────────┼────────────────┐
                          ▼               ▼                ▼
                    ┌─────────┐     ┌─────────┐      ┌──────────┐
                    │ Webhook │     │  Email  │      │ History  │
                    │ Sender  │     │ Sender  │      │  sink    │
                    └─────────┘     └─────────┘      └────┬─────┘
                                                          ▼
                                                  ┌───────────────┐
                                                  │  alert_event  │
                                                  │  (SQLite)     │
                                                  └───────────────┘
```

- **Storage**:
  - BoltDB buckets `alerting_channels`, `alert_rules`
    (operator config, no secret encryption at-rest beyond
    file permissions 0o600).
  - SQLite `alert_event` table (30 d retention via the
    existing observability pruner).
- **Auth posture**: every `/settings/alerting/*` route is
  admin-only via the existing admin sub-group. The
  read-only `/observability/alert-events` is viewer+.
- **Lifecycle**: watcher starts at the end of boot
  (after observability + cert tracker + health checker
  are wired), logs `alerting watcher started
  interval=30s sources=[...]`. On SIGTERM, drains during
  defer unwind and logs `alerting watcher stopped`.

## Known limitations V1

These are conscious V1 choices, not bugs. Documented at
length in `docs/alerting.md#limitations-v1`.

- **Source picker hardcoded** — 3 sources only
  (`waf_event_rate`, `cert_expiry`, `system_health`).
  Registry is static at boot. V2 if more sources land.
- **No retry on send failure** — single attempt per
  dispatch. Receiver-side reliability assumed.
- **No native Slack / Discord** — use webhook with a
  template body in the receiver's expected JSON shape.
- **Cooldown LRU reset on restart** — in-memory only.
  Mid-incident restart may re-fire all active alerts.
- **No "resolved" notifications** — Arenet fires on
  condition rising, not on condition clearing.
- **Templates sandboxed** — Go `text/template` stdlib
  only, no sprig (env / expandenv / tpl are SSRF +
  RCE surfaces for hand-edited operator templates per
  ADR D8).
- **Single-tenant** — no organization / team concept.
  All admins share the alerting config.
- **Webhook POST only** — no PUT/PATCH (rare in
  practice; V2 if needed).

## Acceptance criteria

- AC #1 — Operator can create a webhook channel via the
  UI and send a synthetic test from the channel row's
  Test button. ✓
- AC #2 — Operator can create an email channel via the
  UI (SMTP host/port/auth/From/To/TLS mode) and send
  a synthetic test. ✓
- AC #3 — Operator can create a rule via the UI with
  one of 3 sources, threshold OR state evaluator,
  multi-channel routing, cooldown 30-86400 s. ✓
- AC #4 — Watcher polls every 30 s and persists
  `LastEvalAt` / `LastFiredAt` / `LastError` on every
  rule row. Visible in the UI via the Règles table. ✓
- AC #5 — Cooldown LRU prevents alert spam: a re-fired
  condition within the cooldown window does NOT send a
  duplicate notification, but the heartbeat
  `LastEvalAt` still updates. ✓
- AC #6 — Every dispatch (whether from watcher or from
  /test) persists one `alert_event` row visible in the
  History tab. ✓
- AC #7 — Operator can filter History by date range,
  severity, rule, category. ✓
- AC #8 — Backend degraded-mode contract preserved:
  `obsStore == nil` makes the dispatcher skip the
  history sink + makes `/alert-events` return
  `{events:[], degraded:true}` instead of 5xx. ✓
- AC #9 — All write endpoints emit audit events
  (`alert_channel_*` × 3, `alert_rule_*` × 3). ✓
- AC #10 — Secret redaction: webhook header values
  appear as `[redacted]` on GET and in audit;
  SMTP password appears empty on GET + preserve-on-omit
  on PUT. ✓

## Migration / upgrade

No migration required. The `alert_event` SQLite table
was provisioned in AL.1.a (schema v9→v10) but had no
producer until AL.4.a; existing arenet upgrades simply
start populating the table on next boot once the new
binary is in place.

No new BoltDB migration: the two new buckets
(`alerting_channels`, `alert_rules`) are
`CreateBucketIfNotExists`-style — fresh installs and
upgrades both work the first boot.

No new environment variables. No new flags.
